# Skills

## このプロジェクトで必要なスキル

「勇者のあとしまつ / ダンジョン清掃員の生活」を開発するために必要なスキルをまとめる。

最初からすべてを作り込まず、MVPの小さなループを完成させながら段階的に伸ばしていく。

## Phaser.js / TypeScript

- PhaserのScene、GameObject、Physics、Camera、Inputを理解する
- `BootScene`、`TitleScene`、`BaseScene`、`DungeonScene`、`ResultScene` の役割を分ける
- TypeScriptでプレイヤー、敵、残骸、素材、アップグレードの型を定義する
- ゲームロジックと表示処理を混ぜすぎない
- Phaserの更新ループで重い処理を毎フレーム実行しない

## 2Dゲーム設計

- 見下ろし型マップ、壁、床、出口、残骸、素材、罠、敵を配置する
- プレイヤーの移動、当たり判定、カメラ追従を実装する
- 掃除アクションの手応えを作る
- 敵は倒す対象ではなく、作業を邪魔する危険要素として扱う
- 1回3〜5分で終わるテンポを維持する

## スマホUI / 入力設計

- スマホ縦画面を優先する
- 左下に固定仮想スティックを置く
- 右下に掃除の大ボタンと回避の小ボタンを置く
- 「調べる」は常時ボタンにせず、状況アクションとして掃除ボタンを切り替える
- 素材は近づいたら自動回収する
- タップ範囲を広めに取り、押し間違いを減らす
- 主要ボタンは64px以上、その他のタップ領域は48px以上を目安にする
- タッチ操作中のブラウザスクロールを防ぐ
- セーフエリアを考慮して、重要操作を画面端に寄せすぎない
- 画面スクロールやブラウザジェスチャーの誤作動を防ぐ
- PCではWASD、方向キー、Space、Shift、E、Escを使えるようにする

## 掃除アクション設計

- 掃除は長押し操作にする
- 長押し中は進捗リング、短い揺れ、粉じん、掃き音で手応えを出す
- 長押しを離したら掃除を中断する
- 残骸の耐久値が0になったら素材をドロップする
- 耐久値が低い残骸はすぐ消し、耐久値が高い残骸は複数拍かけて消す
- 対象選択はプレイヤーの向き、または近くの残骸を優先する

## PWA / Webアプリ化

- `manifest.json` を設定する
- アイコン、テーマカラー、画面向き、表示モードを整える
- Service Workerで基本ファイルをキャッシュする
- ホーム画面追加とオフライン起動を確認する
- スマホブラウザの音声自動再生制限を考慮する

## localStorageセーブ

- 所持金、所持素材、ほうきレベル、バッグレベル、到達ダンジョン、設定情報を保存する
- 保存データには `version` を持たせる
- リザルト表示時、強化完了時、設定変更時、拠点に戻った時に保存する
- 壊れたセーブデータを読んでもゲームが停止しないようにする
- 将来のクラウドセーブ導入を見越して、保存形式を単純に保つ

## Cloudflare Pages

- 静的ビルド成果物をCloudflare Pagesで公開できるようにする
- MVPではサーバー、DB、認証、ランキングを使わない
- 将来サーバーが必要になった場合はCloudflare Workers、Supabase、Firebase、Cloudflare D1を検討する
- 公開前にスマホ実機、PC、ブラウザサイズ変更を確認する

## ゲームバランス

- ほうき強化は清掃効率に直結させる
- バッグ強化は持ち帰れる素材量に直結させる
- 報酬は分かりやすく、次の強化目標が見えるようにする
- 敵や罠は強くしすぎず、作業中の緊張感を作る程度にする
- 「もう一回だけダンジョンに行きたい」と感じる短い成長ループを優先する

## ドット絵 / アセット管理

- 粗めのドット絵で軽量に作る
- 背景はタイルベースにする
- 光、影、ぼかし、パーティクルは控えめにする
- 画像と音声はファイルサイズを抑える
- 仮素材でも、用途と差し替え予定が分かる名前にする

## テスト観点

- タイトルから拠点、ダンジョン、リザルトへ進めるか
- スマホで移動と掃除がしやすいか
- 移動しながら掃除ボタンを押し続けられるか
- 掃除、回避、状況アクションを押し間違えにくいか
- PCキーボードで操作できるか
- 壁をすり抜けないか
- 残骸を掃除して素材を拾えるか
- 出口から帰還できるか
- 所持金と素材が増えるか
- 強化できるか
- セーブと続きから開始が動くか
- ビルド後もCloudflare Pagesで公開できるか
